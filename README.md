## 1. AGENTS.md とは（まず一言で）

ひとことで言うと、**AGENTS.md は「AIエージェントに向けて書く、プロジェクトの取扱説明書」**です。公式サイトはこう表現しています——「**AGENTS.md は README for agents（エージェント向けのREADME）だと考えてください**」。✅

README.md は「人間の開発者」に向けて、プロジェクトの概要・使い方・貢献方法を書く場所でした。でも AI に作業を任せる時代になると、AI が知りたいことは少し違います。たとえば——

- このプロジェクトは**どのコマンドでビルド・テストするのか**
- **コードの書き方のルール**（命名・フォーマット・使っていいライブラリ）は何か
- **やってはいけないこと**（本番にpushしない、このディレクトリは触らない 等）

これらを README にぜんぶ詰め込むと、人間にとっては読みにくくなります。そこで「**人間向けは README.md、AI向けは AGENTS.md**」と置き場所を分ける、というのが基本の発想です。✅

ファイルの正体は**ただのMarkdown**です。特別な書式・必須項目はありません。普通のMarkdownで「お願いしたいこと」を書けば、対応エージェントがそれを読みます。✅

> 比喩で言うと、AGENTS.md は「**新しく入ったメンバーへの引き継ぎメモ**」です。新人（＝AI）が初日に迷わないよう、「動かし方・ルール・地雷」を1枚にまとめておく。毎回口頭で説明する代わりに、その紙を渡すだけで済むようにするイメージです。

## 2. なぜ今 重要なのか（3本柱）

### 柱① 「設定ファイル乱立」を1枚に収束できる ✅

少し前まで、AIコーディングツールごとに設定ファイルがバラバラでした（`.cursorrules`、各ツール独自の指示ファイル…）。チームで複数のツールを使うと、同じルールを何度も書き直すことになります。

AGENTS.md は**ベンダー中立の1ファイル**なので、対応する多くのツールが同じ指示を読みます（対応ツールは公式記載 ✅）。これにより「**1回書けば多くに効く**」状態を目指せます（実際の効果は使い方しだい 🔶）。

### 柱② AIの「出力の一貫性」が上がり、手戻りが減る 🔶

毎回チャットで「テストはこう動かして」「この規約で書いて」と説明していると、会話が長くなるほどブレます。プロジェクト直下に AGENTS.md があると、エージェントは**作業の起点で毎回それを読む**ので、指示の前提が揃います。

現場（X）でも「**出力の一貫性が良くなった**」「**チーム開発や会話が続く場面で効く**」という声が見られます（⚠️ 個人の感想・効果には差があります）。

### 柱③ オープン標準なのでロックインされない ✅

AGENTS.md は特定企業の製品ではなく、**複数ベンダーの協調で生まれたオープンな取り決め**です。だから「このツールをやめたら設定が無駄になる」というロックインが起きにくい。ツールを乗り換えても AGENTS.md はそのまま使い回せます。

## 4. 中身：何を書くのか（推奨セクション）

必須項目はありませんが、公式や実運用でよく使われる“定番の見出し”があります。✅

- **Project overview**（このプロジェクトは何か・構成）
- **Build and test commands**（ビルド・テストの具体コマンド）
- **Code style guidelines**（命名・フォーマット・規約）
- **Testing instructions**（テストの回し方・通すべきチェック）
- **Security considerations**（触ってはいけない範囲・秘密情報の扱い）
- **Commit message / PR guidelines**（コミット・プルリクの作法）

公式の**サンプル AGENTS.md**は、最小構成として次の3ブロックを示しています：✅

1. **Dev environment tips**（開発環境のヒント：パッケージ間の移動、新規プロジェクトの作り方など）
2. **Testing instructions**（テスト手順：CIの回し方、lintチェック）
3. **PR instructions**（プルリクの作法：タイトル形式、コミット前の必須チェック）

ポイントは「**AIが手を動かすのに必要な“操作情報”を具体的に書く**」こと。概念的な説明より、「このコマンドを叩く」「このチェックを通す」という実行可能な指示が効きます。✅

## 5. 動く仕組み：ネスト・優先順位・読み込み

### モノレポはAGENTS.mdを“入れ子”にできる ✅

大きなリポジトリ（モノレポ）では、**パッケージごとに AGENTS.md を置けます**。公式の説明はこうです——「**各パッケージの中に別の AGENTS.md を置く。エージェントはディレクトリツリーの中で“最も近い”ファイルを自動的に読むので、いちばん近いものが優先される**」。✅

実例として、**OpenAI 自身のリポジトリには 88 個の AGENTS.md ファイル**が置かれているとされています。🔶（公式サイト記載・数は変動しうる）

### 優先順位の原則 ✅

- ルート → 下層へとマージされ、**編集する場所に近い AGENTS.md ほど優先**（後勝ち）。
- そして最上位は**人間の明示的な指示**：チャットで直接出した指示は AGENTS.md の内容より優先される。

### 参考：Codex の読み込み挙動（ツール固有） ✅

ツールによって細部の挙動は違いますが、たとえば Codex の場合は公式ドキュメントで次のように定義されています：

- 読み込み順：グローバル（`~/.codex`：`AGENTS.override.md` があればそれを、無ければ `AGENTS.md` を読む）→ プロジェクト（Gitルートから作業ディレクトリまで、各階層で同じく override 優先）。
- マージ：ルートから下へ連結し、**近いファイルほど後勝ち**。
- 制約：1ファイルの既定サイズ上限は **32 KiB**、空ファイルはスキップ。

→ つまり「**長く書きすぎない**」のは作法であると同時に、ツールによっては**サイズ上限**という実利的な理由もあります。

## 6. 使い方（初心者基準・コピペ可）

ここはとにかく簡単です。**「プロジェクトのいちばん上に AGENTS.md という名前のファイルを作って、お願いを書くだけ」**。

### 手順（3ステップ）

1. プロジェクトのルート（README.md がある場所）に `AGENTS.md` を新規作成する。
2. 「ビルド/テストのコマンド」「守ってほしいルール」「やってほしくないこと」を書く。
3. 対応エージェントに作業を頼む（多くのツールは**自動で読む**ので、特別な操作は不要）。

### こんな悩み → AGENTS.md にこう書く

| あなたの悩み | AGENTS.md に書くこと |
|---|---|
| AIが毎回テストの動かし方を間違える | 「テストは `npm test` で実行。push前に通すこと」 |
| 勝手に余計なライブラリを足される | 「新しい依存は追加しない。標準ライブラリで実装する」 |
| コードのスタイルがバラバラになる | 「インデントは2スペース。命名は camelCase」 |
| 触ってほしくない場所をいじられる | 「`/legacy` 配下は変更禁止」 |
| コミットメッセージが雑 | 「コミットは `feat:` `fix:` の接頭辞を付ける」 |

### そのままコピペできる “最小 AGENTS.md”

```markdown
# AGENTS.md

## このプロジェクトについて
個人ブログのフロントエンド（Next.js）。

## 開発環境
- セットアップ: `npm install`
- 開発サーバー: `npm run dev`

## テスト
- 実行: `npm test`
- 変更を出す前にテストを通すこと。

## コード規約
- インデントは2スペース、命名は camelCase。
- 新しい依存ライブラリは勝手に追加しない。

## やらないこと
- `/legacy` 配下は変更しない。
- `.env` などの秘密情報をコミットしない。
```

> 💡 コツ：最初から完璧を目指さなくてOK。**「AIがやらかしたこと」を見つけるたびに1行ずつ足していく**と、自然と“効く”AGENTS.md に育ちます。

## 7. Claude Code でどう使うか（重要）

ここは多くの人がつまずくポイントです。**Claude Code は AGENTS.md を“そのまま”は読みません**。Claude Code が読むのは `CLAUDE.md` です。✅

> ⚠️ つまり、AGENTS.md だけ置いても Claude Code には効きません。橋渡しが必要です。

### 橋渡しの2つの定石 ✅

**方法A：インポート（推奨）** — `CLAUDE.md` の中で AGENTS.md を読み込む。Claude Code は読み込み時に `@AGENTS.md` を展開するので、「共通ルール（AGENTS.md）＋ Claude 専用の数行」を両立できる。

```markdown
# CLAUDE.md
@AGENTS.md

## Claude Code 専用メモ
- ここに Claude 固有の補足を数行だけ書く。
```

**方法B：シンボリックリンク** — `CLAUDE.md` を AGENTS.md への symlink にする。いちばん単純で、Claude Code はリンク先を自分のファイルとして扱う。

```bash
ln -s AGENTS.md CLAUDE.md
```

### 他ツールとの違い 🔶

一方で **Cursor・Windsurf・Cline などはプロジェクト直下の AGENTS.md を“ネイティブに”直接読む**ようになっています（symlinkや追加設定なしでOK）。Claude Code での「ネイティブ対応」は本資料作成時点で**未実装**で、機能要望（[anthropics/claude-code#34235](https://github.com/anthropics/claude-code/issues/34235)）が立っている状況です。✅🔶（今後ネイティブ対応が入る可能性はあるため要追跡）

## 8. もう一歩：効く書き方のコツと“罠”

### コツ：最小主義 ✅🔶

「長く詳しく書けば賢くなる」わけではありません。現場の主流は**最小主義**——ビルド/テストコマンド、禁止事項、役割分担に絞って簡潔に書く、というスタイルです。理由は2つ：

- **ツールによってサイズ上限がある**（Codex は 32 KiB）。✅
- 書きすぎると、AIが**指示に律儀に従いすぎて柔軟性を失う**（後述の“罠”）。⚠️

### 罠：書きすぎると逆効果 ⚠️

X 上の感想では、「**最初に“良いもの”を書くのが難しい**」「**書きすぎると AI が律儀すぎて、かえって動きが悪くなる**」のが最大の落とし穴、という声が見られます（⚠️ 個人の感想ベース・効果には差）。

→ 対策：いきなり大作を書かず、**「最小から始めて、問題が起きた箇所だけ足す」**運用が安全です（§6のコツと同じ）。

### コツ：実行可能な情報を優先 ✅

「丁寧に書く」より「**AIが手を動かせる具体情報を書く**」。曖昧な理念より「このコマンド」「このチェック」。テスト/lint コマンドを明記しておくと、エージェントはそれを実行して自己検証してくれます。✅

## 9. README.md / CLAUDE.md / .cursorrules との違い

| 観点 | README.md | AGENTS.md | CLAUDE.md | .cursorrules |
|---|---|---|---|---|
| 主な読み手 | 人間 | AIエージェント全般 | Claude Code | （主に）Cursor |
| 立ち位置 | プロジェクト紹介 | **ベンダー中立の共通標準** | Claude Code 固有 | ツール固有（旧来型） |
| 対応範囲 | — | 20以上のツール ✅ | Claude Code | 主に Cursor |
| 形式 | Markdown | Markdown（必須項目なし）✅ | Markdown | 独自/テキスト |
| 関係 | AI向け詳細を分離 | これを“正典”に置く | `@AGENTS.md` で共有可 ✅ | AGENTS.md へ移行が進む 🔶 |

ざっくり整理すると——**「人間向け＝README、AI共通＝AGENTS.md、Claude固有の上乗せ＝CLAUDE.md」**。`.cursorrules` のようなツール固有ファイルは、AGENTS.md への集約が進んでいます。🔶

## 10. よくある質問（FAQ）

**Q. 必須の項目はある？** いいえ。**普通のMarkdownでOK**で、決まったフィールドはありません。✅

**Q. README.md は要らなくなる？** いいえ。人間向けのREADMEはそのまま。AGENTS.md は**AI向けの情報を分けて置く**ためのもので、役割が違います。✅

**Q. AGENTS.md に書いたテストは自動で実行される？** 対応エージェントは、ファイルに書かれたテスト/チェックのコマンドを**実行して検証**できます（ツールによる）。✅

**Q. モノレポでルールが場所ごとに違う場合は？** パッケージごとに AGENTS.md を置けば、エージェントは**最も近いファイル**を優先して読みます。✅

**Q. 既存の `.cursorrules` などからどう移行する？** 内容を AGENTS.md に集約し、ツール固有ファイルは段階的に減らすのが流れ。Claude Code だけは `@AGENTS.md` インポート/symlink で橋渡しする（§7）。🔶

**Q. チャットでの指示と AGENTS.md、どっちが強い？** **人間の明示的な指示が最優先**。AGENTS.md は“常設のお願い”、チャットは“その場の上書き”という関係です。✅
